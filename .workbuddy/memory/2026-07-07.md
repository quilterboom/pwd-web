## 构建 passwdpm-web：服务端 GPG/SM2 加解密密码管理器

完成了一个完整的全栈密码管理工具（Python 后端 + 原生前端）。核心特性：加解密全部在服务器完成、每条记录可选 GPG 或 SM2、密码查看/新增/修改/删除、修改记录（审计日志）。

### 关键技术决策（踩坑记录，重要）
- **运行环境**：使用托管 Python `C:\Users\12580\.workbuddy\binaries\python\versions\3.13.12`，venv 在 `...\envs\passwdpm`。后端依赖装在该隔离 venv。
- **pgpy 在 Python 3.13 不可用（`imghdr` 被移除）**：在 `backend/app/crypto/gpg_crypto.py` 顶部注入 `imghdr` 垫片（`sys.modules['imghdr']`）后再 import pgpy。常量名是 `SymmetricKeyAlgorithm`（非 `SymAlgorithm`）。
- **gmssl SM2 公钥推导**：`CryptSM2` 不会自动由私钥生成公钥。用 `c._kg(int(priv,16), c.ecc_table['g'])` 算 Q=priv*G。加密结果 base64 存储。
- **passlib + bcrypt 不兼容**：passlib 1.7.4 与 bcrypt>=4.1 冲突（72 字节检测报错）。必须固定 `bcrypt==4.0.1`（已写进 requirements.txt）。
- **run.py 入口**：用绝对导入 `from app.config import ...`，从 `backend/` 目录运行 `python run.py`（不能用相对导入，否则 "attempted relative import with no known parent package"）。
- **FastAPI 0.139 的 include_router**：被包成 `_IncludedRouter`，`app.routes` 不直接展开 API 路径；需用 TestClient 或真实请求验证，不能只看 routes 列表。
- **密钥/数据库**：首次启动 `seed.seed()` 自动建表、建管理员（env ADMIN_USERNAME/ADMIN_PASSWORD，默认 admin/admin123）、生成 gpg+sm2 密钥对存入 SQLite（`backend/data/passwdpm.db`）。JWT 密钥写入 `backend/.secret_key` 持久化（已 gitignore）。

### 新增功能：文件保险箱（File Vault）
- 用户需求：除密码文本外，要能对「文件」加密（参考 D:\aicode\passwdpm 即 PassPy 的 GPG/SM2 文件加密过程）。已新增。
- 复用服务端 GPG/SM2 密钥：上传文件 → 服务端用所选算法公钥加密落盘（密文存 `backend/data/files/<id>.<gpg|sm2>`，DB 只存元数据）；可「下载密文」/「解密下载原文」；解密/上传/删除记审计（`FileHistory` 表）。
- 加密实现：gpg_crypto 加 `encrypt_bytes/decrypt_bytes`（pgpy PGPMessage 原生支持二进制，内部即混合加密）；sm2_crypto 加 `encrypt_bytes/decrypt_bytes`（gmssl CryptSM2.encrypt 支持任意长度，KDF 思路）。manager 加 `encrypt_file/decrypt_file`。
- 新增 `backend/app/routers/files.py`（/api/files: upload/list/download/decrypt/delete/history）；models 加 `FileVault`+`FileHistory`；config 加 `FILES_DIR`；main.py 注册路由。
- 前端：index.html 加「密码 / 文件保险箱」页签 + 文件面板（上传按钮+算法下拉+表格）；app.js 加 tab 切换、uploadFile、downloadCipher/decryptDownload（fetch+blob 触发下载，解析 Content-Disposition 的 filename* 中文）、openFileHistory、deleteFile；styles.css 加 .tabs/.tab/.inline-label。
- **验证**：TestClient 冒烟全过——GPG/SM2 上传、密文≠明文、解密==原文、history 记录 upload+decrypt、删除后列表减少、未授权 401。
- 设计取舍：文件也加密到「服务端密钥」（与密码记录一致的模型），非用户自带密钥；SM2 直接加密（未做 SM4 混合），密码管理器场景文件通常较小，够用；删除为软删除（密文文件保留在磁盘）。

### 新增功能：多账号 + 分组隔离（Multi-Tenant by Group）
- 用户需求：能**新增账号**，且页面看到的数据与「分组」绑定——普通用户只看到自己所属分组的数据；管理员可跨组查看全部。
- 数据模型：`User`(加 is_admin)、`Group`、`user_groups` 多对多关联表；`PasswordEntry`/`FileVault`/`History`/`FileHistory` 均加 `group_id`。
- 权限依赖 `backend/app/core/deps.py`：`get_user_groups`（管理员=全部分组，普通=所属分组）、`get_user_group_ids`、`visibility_filter`（管理员返回 None 看全部，普通 `column.in_(ids)`）、`ensure_group_access`（写入/读取时校验分组归属）、`require_admin`。
- **关键坑：SQLAlchemy 多对多没有 `UserGroup` 模型类**，只有 `user_groups` Table 对象。所有地方必须用 `user_groups.insert()/.select()/.delete()`，不能用 `UserGroup(...)` ORM 对象（会 ImportError）。已统一修正 admin.py/deps.py/seed.py 与测试脚本。
- 管理接口 `backend/app/routers/admin.py`：拆成 3 个 router——`mine_router`(/api/groups/mine)、`users_router`(/api/admin/users CRUD+分组)、`groups_router`(/api/admin/groups CRUD+成员)。main.py 注册全部三个。
- 分组删除保护：仍绑定密码/文件的分组拒绝删除（409/400），避免数据“消失”。
- 平滑迁移 `db._migrate_columns()`：旧库 ALTER 加 `users.is_admin`、`*.group_id` 列（新表 groups/user_groups 由 create_all 自动建）。`seed()` 额外把 NULL group_id 的存量数据归到默认分组，防止升级后旧数据对普通用户不可见。
- config.py 现支持 `DATA_DIR` 环境变量（默认 backend/data），便于测试隔离与部署时重定向数据目录。
- 前端：管理员可见「⚙️ 管理」入口，弹窗内「账号 / 分组」子页签做 CRUD + 成员分配；新增密码/文件时必须选所属分组（下拉来自 /api/groups/mine）；列表增加「分组」列；表格/编辑回填分组名。
